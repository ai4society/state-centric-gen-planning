# Data Directory Documentation

This directory contains the complete lifecycle of planning data, ranging from raw PDDL definitions to machine-learning-ready graph embeddings. The data is organized to support **State-Centric Generalized Planning**.

## Directory Structure

```text
data/
├── pddl/                  # Raw Domain and Problem definitions
│   ├── <domain>/          # e.g., blocks, gripper, logistics
│   │   ├── domain.pddl
│   │   ├── train/
│   │   ├── validation/
│   │   ├── test-interpolation/
│   │   └── test-extrapolation/
├── plans/                 # SAS+ Plans generated by Fast Downward
│   ├── <domain>/
│   │   └── <split>/       # Matches pddl/ structure
│   │       └── <problem>.plan
├── states/                # State Trajectories (sequences of predicates)
│   ├── <domain>/
│   │   └── <split>/
│   │       └── <problem>.traj
└── encodings/             # ML-Ready Vector Representations
    └── graphs/            # Fixed-size WL Graph Embeddings
        ├── <domain>/
        │   └── <split>/
        │       ├── <problem>.npy       # Trajectory embedding matrix [T, D]
        │       └── <problem>_goal.npy  # Goal embedding vector [D]
```

## 1. PDDL (`data/pddl/`)

Contains the symbolic definitions of the planning tasks. There are taken from the [Symmetry-Aware Transformer Training for Automated Planning Repository](https://github.com/mrlab-ai/symmetry-transformers/tree/main) ([permalink when copied](https://github.com/mrlab-ai/symmetry-transformers/tree/56983a26a37e28e2f148ce067565c9d8eefb85eb/pddl)). Some processing was done, such as lowercasing.

- Format: Standard PDDL (Planning Domain Definition Language).
- Splits:
  - `train`: Small instances (e.g., 2-5 blocks) used for learning.
  - `validation`: Slightly larger instances for hyperparameter tuning.
  - `test-interpolation`: Instances of similar size to training but unseen configurations.
  - `test-extrapolation`: Large instances (e.g., 10+ blocks) to test Out-of-Distribution (OOD) generalization.

## 2. Plans (`data/plans/`)

Contains the optimal (or satisficing) sequence of actions to solve the corresponding PDDL problems.

- Generator: Fast Downward (A\* with `lmcut` or Greedy Best First Search).
- Format: One action per line, lowercase, parenthesized.
  ```lisp
  (pick-up b)
  (stack b a)
  ```
- Correspondence: `data/plans/<dom>/<split>/<prob>.plan` corresponds to `data/pddl/<dom>/<split>/<prob>.pddl`.

## 3. States (`data/states/`)

Contains the full state trajectory resulting from executing the plan.

- Generator: `VAL` (Plan Validator) replaying the plan against the domain logic.
- Format: `.traj` text files. Each line represents a discrete time step $t$. The line contains a space-separated list of all **true predicates** in that state (Closed World Assumption).
  ```lisp
  (on a b) (on b c) (clear a) (ontable c) ...
  (holding a) (on b c) (clear b) (ontable c) ...
  ```
- Note: Line 0 is the Initial State ($S_0$). The last line is the Goal State ($S_g$).

## 4. Encodings (`data/encodings/graphs/`)

Contains numerical vector representations of the states and goals, generated using the **Weisfeiler-Leman (WL)** algorithm on **Instance Learning Graphs (ILG)**.

- Generator: [`wlplan` library](https://github.com/DillonZChen/wlplan).
- Vocabulary Consistency: The feature mapping (hashing of graph substructures to vector indices) is learned **only** from the `train` split but applied consistently to `validation` and `test`.
- Files:
  1.  `problem_name.npy`: A NumPy matrix of shape `(T, D)`.
      - $T$: Number of time steps in the trajectory.
      - $D$: Dimension of the WL feature vector (vocabulary size).
      - Represents the sequence of states $S_0 \dots S_g$.
  2.  `problem_name_goal.npy`: A NumPy vector of shape `(D,)`.
      - Represents the embedding of the Goal Condition $G$ as a graph.
      - Essential for conditioning generative models (e.g., $P(S_{t+1} | S_t, G)$).
